# AIOps 黑客松 · 方向收敛

## 0. TL;DR

**押候选 1（故障定位 Agent），切口从「变更归因」上移到「私有化因果链事故调查员」；翻车则退候选 2 或候选 3。**

决策双判据（缺一不可）：

1. **技术判据**：今晚 SRE 队友能否造出"像真的"K8s 级联故障遥测数据 —— 能则锁候选 1，不能则退候选 3（数据天然结构化、demo 更稳）。
2. **创新性判据**（评分表场景创新性占 30%）：demo 必须能跑出"因果链至少一跳"（变更 → 顺拓扑/调用链 → 命中级联放大点），否则纯"变更归因"会被评委判为"换壳"（Azure SRE Agent / Rootly / Harness / Datadog 已是现成功能）。

## 1. 比赛背景

- **主题**：ToB 场景 AI Agent（开放赛道）
- **核心评判**：AI Agent 能否解决企业**真问题**（不是又一个工具/demo，要有真实痛点 + 可量化 ROI）
- **时间**：2026-06-27 11:00 → 2026-06-28 14:30（约 27 小时）
- **举办方**：微软孵化器 + 小宿科技（[xiaosuai.com](https://www.xiaosuai.com/)，AI Agent 基础设施一站式服务商：智能搜索 API / SkyRouter 模型聚合 / GPU 云 / AI 沙盒 / 多云管理）
- **团队不公平优势**：队友 = 前云 SRE，现 ToB AI 架构师 → 真 SRE 做 SRE Agent，团队-问题匹配可信度拉满

### 评分标准（2026 黑客松实际权重）

| 维度 | 权重 | 评委重点 |
|---|---|---|
| 场景创新性 | **30%** | 问题价值 + 切入新意 |
| 产品完成度 | 25% | 真实可运行、核心流程完整 |
| 技术深度 | **20%** | 架构合理 + 工程思考 |
| 商业潜力 | 15% | 用户/场景/落地空间 |
| Demo 表现 | 10% | 表达清楚 + 现场稳定 |

> 高权重在创新性（30%）+ 技术深度（20%）= 50%，方案设计须优先满足这两块。

## 2. 主推方案 🥇 私有化因果链事故调查员

### 2.1 定义与工作流

告警一响，Agent 自主开干：拉日志、查指标、看最近部署/变更、**顺拓扑/调用链做因果推理**、形成根因假设、给出（或一键执行）止血动作、自动写事故时间线。每一步推理带置信度 + 可审计证据链，全程数据不出域。

与"纯变更归因"的关键差别：Agent 不止回答"哪次变更搞的"，而是**顺着拓扑/调用链推一跳** —— "这次变更 → 如何级联放大 → 命中哪个服务雪崩"。

### 2.2 差异化三件套（开场 30 秒必讲）

1. **跨信号因果推理**（不是 join）；
2. **每步可验证证据链 + 置信度**（攻 agent 幻觉真空，运维对错误零容忍）；
3. **数据不出域私有化**（对标主办方小宿，金融/政企刚需，竞品结构性做不到）。

> 这条"慢 SQL/锁等待 → 连接池打满 → 服务级联雪崩"的因果级联叙事，正是创新性内核的来源（与 Dynatrace Davis / Traversal 的因果 + 拓扑前沿同构）。完整 DB 遥测数据成本高、可砍，只保留"一跳因果"的最小切片。

### 2.3 Demo 脚本

改一个 K8s configmap → 服务 500 → Agent 顺调用链定位变更 + **推出级联放大点** + 一键回滚恢复。

**现场杀手锏**：让评委现场指定/改一个变量 —— 破"固定路径"质疑、补技术深度分。

### 2.4 落地故障场景：K8s 崩溃专科

主线因果链需要落到一类具体故障上演示，选 **CrashLoopBackOff / OOMKilled**：Agent 读 pod events + logs + resource limits + 最近镜像变更 → 给根因 + 修复建议。

> ⚠️ **真实感提示**：真实 OOM events 是脏的、滞后的、和 node pressure 缠在一起；造的数据太干净会被懂行评委看穿，注意做"脏一点"。

## 3. 备选方案（翻车退路）

### 3.1 候选 2 🥈 变更风险审查 Agent（发布/IaC 守门员）

Agent 审 Terraform / K8s / 发布 PR，预测爆炸半径、标红高危变更、给出回滚预案，人审确认后放行。

### 3.2 候选 3 🥉 LLMOps 哨兵（运维 AI 应用本身）

别人都在用 AI 做运维，谁来运维 AI？接入 LLM 应用的调用记录（prompt / 回复 / 延迟 / token / 成本），用 LLM-as-judge 自动检测幻觉、质量漂移、成本异常，并给出 prompt 修复建议。

**为什么是最稳退路**：数据天然结构化、不用造故障，demo 最稳，且正中小宿 Agent 客户的痛点。

---

## 4. 范围控制（27 小时纪律）

> ⚠️ **不要为"丰富"把候选 1/2/3 并成三条独立路径演示** —— 那是范围蔓延和虚荣指标。
>
> 27 小时只打磨**一条**"变更 → 根因推理 → 止血"的路径：跑通、跑稳、能现场改变量。三个方案共用同一套差异化内核（根因推理 + 置信度证据链 + 人工把关 + 数据不出域），主攻主线即可。

## 附录：AIOps 市场调研底稿

> 以下为方向收敛前的原始调研，正文结论由此提炼，保留作留痕。

### 一、正在快速落地的几个具体方向

1. **LLM 重做可观测性（Observability）交互层**
   - 自然语言查日志/指标/trace，取代 PromQL/SQL 的门槛
   - 自动生成事故时间线、postmortem、值班交接摘要
   - 和 OpenTelemetry 标准深度绑定
2. **根因分析（RCA）的多模态融合**
   - logs + metrics + traces + 部署/变更事件 关联
   - 拓扑感知 + 因果推断，而非单纯统计相关
3. **事件管理（Incident Management）智能化**
   - 自动分级、自动拉群、自动起草沟通稿、推荐 on-call

### 二、刚冒头、但值得押注的反向方向

4. **LLMOps / GenAI 系统自身的可观测性 ——「运维 AI 应用本身」**
   一句话：别人都在用 AI 做运维，谁来运维 AI？接入应用的调用记录（prompt / 回复 / 延迟 / token / 成本），用 LLM-as-judge 自动检测幻觉、质量漂移、成本异常，并自动给出 prompt 修复建议。（→ 即正文候选 3）

### 三、可以切入的缝隙

**缝隙 1：私有化 / 数据不出域（最大机会）**
中国企业核心痛点是"日志/指标是核心资产，绝不能传给 Datadog 这种 SaaS"。巨头的 SaaS 模式天然进不来，而 RCA 又恰恰需要喂大量内部数据。
→ 做"数据不出域的本地化 SRE Agent"，直接对标小宿的 AI 沙盒/私有化算力栈 —— 对着主办方能力讲故事。

**缝隙 2：运维场景的"零幻觉容忍" + 可解释性**
运维对错误零容忍、要求可解释、私有语料不足。巨头的 copilot 还停在"建议"层不敢"执行"，就是因为信任问题。
→ 带置信度 + 证据链 + human-in-the-loop guardrail 的执行，让 Agent 每一步推理都可审计，同时解决"Agent 自主性"和"企业敢用"。

**缝隙 3：垂直收窄，不做通用**
通用平台的死穴是"泛化能力差，跨业务/跨栈迁移难"。
→ 只挑一个具体技术栈的一类故障（K8s 微服务级联故障，或数据库慢查询引发雪崩），做深、做到真实闭环。窄而深 > 宽而浅。
