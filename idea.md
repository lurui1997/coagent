# AIOps 黑客松 · 方向收敛

## 比赛 Context
- **主题**：ToB 场景 AI Agent（开放赛道）
- **核心评判**：AI Agent 能否解决企业**真问题**（不是又一个工具/demo，要有真实痛点 + 可量化 ROI）
- **时间**：2026-06-27 11:00 → 2026-06-28 14:30（约 27 小时）
- **举办方**：微软孵化器 + 小宿科技（[xiaosuai.com](https://www.xiaosuai.com/)，AI Agent 基础设施一站式服务商：智能搜索 API / SkyRouter 模型聚合 / GPU 云 / AI 沙盒 / 多云管理）
- **队伍**：队友 = 前云 SRE，现 ToB AI 架构师 → **不公平优势**：真 SRE 做 SRE Agent，团队-问题匹配可信度拉满

---

**结论:押候选 1(故障定位 Agent),但切口从「变更归因」上移到「私有化因果链事故调查员」;翻车则退候选 2 或候选 3。**

判据(双判据,缺一不可):
1. **技术判据**:今晚 SRE 队友能否造出"像真的"K8s 级联故障遥测数据——能则锁候选 1,不能则退候选 3(数据天然结构化、demo 更稳)。
2. **创新性判据(新增,因评分表场景创新性占 30%)**:demo 必须能跑出"因果链至少一跳"(变更 → 顺拓扑/调用链 → 命中级联放大点),否则纯"变更归因"会被评委判为"换壳"(Azure SRE Agent / Rootly / Harness / Datadog 已是现成功能)。

### 评分表(2026 黑客松实际权重)与候选 1 自评

| 维度 | 权重 | 评委重点 | niche1 原方案(变更归因) | 改后(私有化因果链) |
|---|---|---|---|---|
| 场景创新性 | **30%** | 问题价值 + 切入新意 | 🔴 弱(4 个现成对标) | 🟢 因果+可验证+私有化 |
| 产品完成度 | 25% | 真实可运行、核心流程完整 | 🟡 强但有"固定路径"雷 | 🟡 需现场可变一个变量 |
| 技术深度 | **20%** | 架构合理 + 工程思考 | 🔴 弱("就是个 join") | 🟢 拓扑因果推理 + Agent 编排 |
| 商业潜力 | 15% | 用户/场景/落地空间 | 🟢 强 | 🟢 强 |

> 关键洞察:niche1 最强的两条(商业潜力 15% + Demo 10% = 25%)恰是低权重;最弱的两条(创新性 30% + 技术深度 20% = 50%)恰是高权重。原方案是在为低权重区优化。**改法不是换赛道,是换标题:同一套代码,把叙事重心从"关联"挪到"因果 + 私有化 + 可验证"。**

### 竞争现实(为什么必须改)

| 现成产品 | 做的事 | 与原 niche1 重合 |
|---|---|---|
| **微软 Azure SRE Agent** | "过去一小时改了什么"带引用 | ⚠️ 几乎 100%,且是主办方自家 |
| Rootly AI SRE | 关联代码变更+遥测,根因+置信度+diff | ⚠️ 连证据链都一样 |
| Harness AI SRE | 关联 CI/CD+配置+基础设施变更,根因+爆炸半径 | ⚠️ 高 |
| Datadog Bits AI | 告警→调查→"什么变了" | ⚠️ 高 |
| Parity (YC) | K8s-first 秒级 RCA + runbook | ⚠️ 对标 niche2 |

而 2026 前沿信号一致指向:**差异化在"因果推理 + 拓扑"(Dynatrace Davis、Traversal 90%+ RCA),不在"关联匹配"**。两块尚未被占的真空:① 可验证/可审计的 agent 推理(攻幻觉);② 数据不出域私有化——所有头部玩家(Resolve $250M、Cleric、Traversal、Datadog、Harness)均为美国 SaaS,结构上进不来中国金融/政企,且正中主办方小宿的私有化算力栈。

## 候选方案

### 1. On-call 故障定位 Agent（AIOps 事故调查员）🥇 主推
**对外标题：私有化的因果链事故调查员。**
告警一响，agent 自主开干：拉日志、查指标、看最近部署/变更、**顺拓扑/调用链做因果推理**、形成根因假设、给出（或一键执行）止血动作、自动写事故时间线。每一步推理带置信度 + 可审计证据链，全程数据不出域。

#### 切口定位（创新性优先重排序）

**🥇 主线：因果链事故调查员（变更归因为起点，因果推理为主秀）**
- 是什么：告警一响，Agent 不止回答"哪次变更搞的"，而是**顺着拓扑/调用链推一跳**——"这次变更 → 如何级联放大 → 命中哪个服务雪崩"，给出带置信度的因果链 + 可审计证据 + 止血/回滚建议（人工确认）。
- 为什么这样改：纯"变更归因"已是 Azure SRE Agent / Rootly / Harness / Datadog 的现成功能，在创新性（30%）上等于换壳；**前沿差异化在"因果+拓扑"，且可验证推理 + 私有化是两块美国 SaaS 进不来的真空**。
- 差异化三件套（开场 30 秒必讲）：① 跨信号**因果推理**（不是 join）；② 每步**可验证证据链 + 置信度**（攻 agent 幻觉真空）；③ **数据不出域私有化**（对标主办方小宿，金融/政企刚需，竞品结构性做不到）。
- Demo：改一个 K8s configmap → 服务 500 → Agent 顺调用链定位变更 + **推出级联放大点** + 一键回滚恢复；**让评委现场指定/改一个变量**（破"固定路径"、补技术深度）。

**🥈 niche 2：K8s 崩溃专科（CrashLoopBackOff / OOMKilled，作主线落地的具体故障场景）**
- 是什么：Agent 读 pod events + logs + resource limits + 最近镜像变更 → 给根因 + 修复建议。
- 定位：作为主线因果链落地的**具体技术场景**，不单独成项（已有对标 Parity，单做创新性弱）。
- 风险提示：真实 OOM events 是脏的、滞后的、和 node pressure 缠在一起；造的数据太干净会被懂行评委看穿，注意做"脏一点"。

**🥉 niche 3 的因果链 = 已并入主线**
- "慢 SQL/锁等待 → 连接池打满 → 服务级联雪崩"这条因果链**正是主线创新性的来源**（与 Dynatrace Davis / Traversal 的因果+拓扑前沿同构）。
- 不再作为独立第二场景，而是把它的"因果级联"叙事注入主线；数据成本高的完整 DB 遥测可砍，只保留"一跳因果"的最小切片。

> 三个切口共用同一套差异化：**因果推理 + 置信度证据链 + human-in-loop + 数据不出域**。主攻"因果链事故调查员"主线，niche 2 作具体故障场景落地，niche 3 的因果叙事并入主线作创新性内核。
> ⚠️ 范围警告（精益视角）：不要为"丰富"而把 niche1+2+3 并成三条独立路径演示——那是范围蔓延和虚荣指标。27 小时只打磨**一条**"变更→因果一跳→止血"的路径，跑通、跑稳、能现场改变量。

### 2. 变更风险审查 Agent（发布/IaC 守门员）🥈 备胎
agent 审 Terraform/K8s/发布 PR，预测爆炸半径、标红高危变更、给出回滚预案，人审确认后放行。

### 3. LLMOps 哨兵（运维 AI 应用本身）🥉 退路
别人都在用 AI 做运维，谁来运维 AI？接入 LLM 应用的调用记录(prompt / 回复 / 延迟 / token / 成本)，用 LLM-as-judge 自动检测幻觉、质量漂移、成本异常，并给出 prompt 修复建议。数据天然结构化、不用造故障，demo 最稳，且正中小宿 Agent 客户的痛点。

## AIOps 市场分析

### 一、正在快速落地的几个具体方向
1. **LLM 重做可观测性(Observability)交互层**
   - 自然语言查日志/指标/trace，取代 PromQL/SQL 的门槛
   - 自动生成事故时间线、postmortem、值班交接摘要
   - 和 OpenTelemetry 标准深度绑定
2. **根因分析(RCA)的多模态融合**
   - logs + metrics + traces + 部署/变更事件 关联
   - 拓扑感知 + 因果推断，而非单纯统计相关
3. **事件管理(Incident Management)智能化**
   - 自动分级、自动拉群、自动起草沟通稿、推荐 on-call

### 二、刚冒头、但值得押注的反向方向
4. **LLMOps / GenAI 系统自身的可观测性：LLMOps 哨兵 ——「运维 AI 应用本身」**
   一句话：别人都在用 AI 做运维，谁来运维 AI？做一个监控 LLM 应用质量的平台：接入应用的调用记录(prompt / 回复 / 延迟 / token / 成本)，用一个 LLM 当裁判(LLM-as-judge) 自动检测幻觉、质量漂移、成本异常，并自动给出 prompt 修复建议。

### 三、可以切入的缝隙
**缝隙 1：私有化 / 数据不出域(最大机会)**
中国企业核心痛点是"日志/指标是核心资产，绝不能传给 Datadog 这种 SaaS"。巨头的 SaaS 模式天然进不来。而 RCA 又恰恰需要喂大量内部数据。
→ 做一个"数据不出域的本地化 SRE Agent"，直接对标小宿的 AI 沙盒/私有化算力栈——这是对着主办方能力讲故事。

**缝隙 2：运维场景的"零幻觉容忍" + 可解释性**
研究里反复出现的痛点：运维对错误零容忍、要求可解释、私有语料不足。巨头的 copilot 还在"建议"层不敢"执行"，就是因为信任问题。
→ 差异化：带置信度 + 证据链 + human-in-the-loop guardrail 的执行，让 Agent 的每一步推理都可审计。这恰好把"Agent 自主性"和"企业敢用"两件事同时解决。

**缝隙 3：垂直收窄，不做通用**
通用平台的死穴是"泛化能力差，跨业务/跨栈迁移难"。
→ 只挑一个具体技术栈的一类故障(比如 K8s 微服务的级联故障、或数据库慢查询引发的雪崩)，做深、做到能真实闭环。窄而深 > 宽而浅。
