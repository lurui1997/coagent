# CoAgent 项目介绍
**产品定位：** 面向企业的 Agent 运维助手  
**一句话价值：** Agent 出错时，几分钟内告诉你要不要动、怎么动、有多大把握，并把整个过程记下来让团队越用越准。

**出错**指 Agent 运行态偏离预期，不限于进程崩溃或 HTTP 报错，主要包括：**运行失败**（如 API 429 限流、调用超时）、**质量异常**（如 检索增强 空检索导致幻觉答复）、**成本异常**（如日 Token 超预算）——S1/S2/S3 分别对应三类。

---

## 1. 为什么做这个项目

### 1.1 背景：Agent 进了业务，运维还没跟上

中小企业与 OPC 已将客服、检索增强 问答、内容生成等场景交给 AI Agent 运行，但 **传统基础设施监控管服务器与链路、通用聊天机器人 管问答——中间缺一层「Agent 运行态运维」**。

### 1.2 五大痛点

| # | 痛点 | 业务后果 | CoAgent 对应场景 |
|---|------|----------|-----------------|
| 1 | **静默失败** — Agent 429 / 空检索挂了，客户先发现 | 漏单、客诉 | S1 限流 · S2 空检索 |
| 2 | **不会处置** — 不知先重试、改 提示词 还是换模型 | 翻日志、瞎改配置 | 处置手册+大模型步骤 |
| 3 | **成本失控** — Token 单日超预算无分级响应 | 烧钱、权责不清 | S3 超预算 |
| 4 | **不敢执行** — 不知道 AI 建议靠不靠谱 | 团队不敢用 自主 Agent | 把握度评分 🟢🟡🔴 |
| 5 | **无法复盘** — 说不清当时建议了什么、谁批准的 | 合规风险、经验无法沉淀 | 实时推送时间线 + SQLite + 飞轮 |

### 1.3 市场真空

现有能力大致分三类，各自覆盖 Agent 运行态运维 的一部分，但都无法单独闭环：

| 能力类型 | 通常覆盖什么 | 缺什么 |
|----------|-------------|--------|
| **基础设施监控 / 应用性能监控** | 服务器、网络、5xx、资源指标 | 不管大模型 Agent 失败、检索质量、Token 成本 |
| **SRE / 变更关联工具** | 变更与遥测关联、根因推测 | 不做 Agent 运维层根因推理与可验证评分 |
| **通知 / 协同通道** | 告警推送、即时消息卡片 | 无推理链、无把握度、无处置飞轮 |
| **通用聊天机器人** | 自然语言问答 | 非事件驱动、无处置手册约束、无审计 |

**CoAgent 占位的真空：** 「运维 AI 应用本身」+ 「零幻觉可解释处置」。具体厂商对比见 [§4 竞品分析](#4-竞品分析) 与 [竞品分析调研](./coagent-competitive-analysis.html)。

## 2. 如何解决：方法论（高维视角）

CoAgent 不是「告警 +大模型写建议」的换壳，而是一套 **可验证处置闭环**：

```
听见异常 → 获得处置建议 → 评估把握程度 → 团队复盘迭代
   │              │                │              │
 告警+面板      大模型+重试        把握度评分    飞轮+@升级
```

### 2.1 四层方法论

| 层级 | 方法 | 核心原则 |
|------|------|----------|
| **感知层** |事件回调接入 + Agent 面板 | 失败第一时间「听见」，不靠客户反馈 |
| **推理层** |处置手册约束下的根因推理 | 不是日志关联，是 Agent 运行态根因链（429→不可用、空检索→幻觉） |
| **决策层** | 把握度评分三因子可验证 | 非大模型自报 置信度；数据·手册·推理 公式可追问 |
| **闭环层** | 人工把关 + 审计 + 飞轮 | 🟢敢动 / 🟡需确认 / 🔴升级；全程留痕、👍👎 迭代 |

### 2.2 创新叙事三件套（答辩必讲）

1. **跨信号 Agent 运维推理** —处置手册约束根因链，非基础设施 变更关联  
2. **每步可验证** — 把握度评分分解，评委可追问公式  
3. **处置可闭环** — 建议 + 把握度 + 重试/升级，全程留痕  

### 2.3 价值递进与版本路线

| 阶段 | 能力 | 回答的问题 |
|------|------|-----------|
| **Basic**（当前） | 事件 → 处置手册 → 评分 → 人工处置 | 要不要动？怎么动？有多把握？ |
| **Pro**（12h+） | 运行时纠偏 + 模型切换 + 企业审计 | 能不能自动/半自动修正？谁批准的？ |
| **Ultra**（后期） | 知识图谱 + 历史推演 + 多 Agent 编排 | 历史上类似事故怎么处理？复杂场景谁来做？ |

## 3. 解决方案设计与功能实现

### 3.1 统一处置流水线

```
事件回调 事件 → 场景路由 → 处置手册 →大模型推理 → 把握度评分
    → 管理台（主）→ 飞书 IM（S1）→ 人工反馈 → 飞轮统计
```

**典型耗时：** 8–15 秒（含大模型）；答辩口径「90 秒内拿到结论」含人工切换 Tab。

### 3.2 三场景演示 — 同一内核的三种切面

CoAgent 用 S1/S2/S3 展示 **「敢动手 → 不敢盲动 → 必须升级」** 的处置边界递进，不是三条独立产品。

#### S1 — 客服 Agent API 限流（主线 · 🟢 可执行）

| 维度 | 数据 |
|------|------|
| Agent | cs-bot（客服 Agent） |
| 事件 | `run_fail` + `rate_limit` |
| 错误 | OpenAI 429: Rate limit exceeded for gpt-4o-mini |
|根因推理 | 大促前 concurrent 5→10 → OpenAI 429 → cs-bot thread 全挂 |
| 模拟指标 | 429 次数 **47/5min**，失败率 **38%**，concurrent=**10**（昨日 5） |
| 运维手册 | OPS-101：等待 / 降并发 / 换 key |
| **把握度评分** | **87 🟢 可执行**（目标区间 82–88） |
| 处置 | 标签页三 显示一键重试；飞书卡片同步 |
| 答辩话术 | 「限流是 Agent 配置打满配额——评分 87，可以先重试。」 |

#### S2 — 检索增强 空检索飙升（质量链 · 🟡 需确认）

| 维度 | 数据 |
|------|------|
| Agent | rag-bot（检索增强 客服 Agent） |
| 事件 | `run_fail` + `empty_retrieval` |
| 错误 | Retrieval returned 0 chunks above threshold 0.7 |
|根因推理 | FAQ 更新后索引 lag 1 天 → 空检索率 5%→**35%** → 幻觉答复 → 客诉 **+12** |
| 模拟指标 | `index_version=v2.3`，`kb_last_sync` 滞后 1 天 |
| 运维手册 | OPS-203：rebuild 索引 / 兜底 提示词 |
| **把握度评分** | **72 🟡 需确认**（目标区间 65–75） |
| 处置 | 标签页三 **⚠️ 不建议重试**；须改知识库后确认执行 |
| 答辩话术 | 「同样 run_fail，S1 能重试，S2 不能——评分 定义敢不敢动的边界。」 |

#### S3 — Agent 日成本超预算（成本链 · 🔴 升级）

| 维度 | 数据 |
|------|------|
| Agent | content-bot（内容生成 Agent） |
| 事件 | `cost_report` + `over_budget`（非报错型事件） |
|根因推理 | 新模板 v3-longform 发布 → Token **1.2M/日 (+180%)** → 成本 **¥28.5 / 预算 ¥20** |
| 模拟指标 | top route=`marketing-batch`，`max_tokens=4096` |
| 运维手册 | OPS-305：限流 / 降级模型 / 暂停 batch /升级审批 |
| **把握度评分** | **58 🔴 需升级**（目标区间 50–58） |
| 处置 | 标签页三 显示 **@负责人**；高风险步骤不可自动执行 |
| 答辩话术 | 「不是坏了，是烧钱——评分 58 告诉你不该自己动，该升级。」 |

### 3.3 把握度评分 — 可验证推理

```
总分 = round(100 × (0.35×数据 + 0.35×手册 + 0.30×推理))
```

| 因子 | 含义 | S1 实测 | S2 实测 | S3 实测 |
|------|------|---------|---------|---------|
| **数据** | 数据完备：字段、日志、工具成功率 | ~92% | ~88% | ~85% |
| **手册** | 手册匹配：症状/类型 vs 运维标签 | ~88% | ~75% | ~62% |
| **推理** | 推理一致：假设/步骤 vs 错误/日志 | 校正后 ~75% | ~58% | ~45% |

| 总分 | 分级 | 动作权限 |
|------|------|----------|
| ≥80 | 🟢 可执行 | 建议执行（人工确认高风险步骤） |
| 60–79 | 🟡 需确认 | 请负责人确认 |
| <60 | 🔴 需升级 | @值班 /升级审批 |

### 3.4 已实现功能清单（可演示）

| 模块 | 实现 | 说明 |
|------|------|------|
| 后端 | FastAPI + 异步编排器 | Python 3.11，全链路异步 |
| 配置 | `data/ops_playbooks.json` 单一真相源 | 3 份处置手册 · 禁止按场景写 Python |
| 前端 | 管理台 HTMX 四标签页 | 总览 / 场景 / 把握度评分 / 飞轮 |
| 实时 | 实时推送时间线 | 故障开始 → 工具 → 大模型 → 评分 → 完成 |
| 持久化 | SQLite 故障记录 + 反馈 + 统计 | 全链路审计 |
| 兜底 | 回放只读重放 + 失败不伪造 | 大模型失败走错误态，不读静态假设 |
| 脚本 | `demo.sh` · `calibrate_scores.sh` | 自动化演示与评分校准 |

### 3.5 投资回报叙事（企业采购）

| 价值 | 量化锚点 | 场景证据 |
|------|----------|----------|
| **少漏单** | 429 失败率 38% → 分钟级发现 | S1：47 次/5min 限流 |
| **少瞎改** | 空检索 35% 时禁止盲重试 | S2：不建议重试 |
| **少烧钱** | 超预算 42% 时强制升级 | S3：¥28.5 vs ¥20 |
| **控得住** | 把握度评分三因子可追问 | 标签页三 数据·手册·推理 分解 |
| **查得着** | 全链路时间线 持久化 | SQLite + 回放 |
| **越用越准** | 👍/👎 驱动手册迭代 | 标签页四 飞轮 |

## 4. 竞品分析

### 4.1 能力类型差异（一句话）

| 他们 | CoAgent |
|------|---------|
| 管服务器 / 5xx | 管 **Agent 失败、质量、成本** |
| 告警 + 建议 | **评分可验证 + 审计闭环** |
| 云服务 吃数据 | **本地部署，日志不出域** |

### 4.2 代表产品与错位竞争

| 代表产品 | 做什么 | CoAgent 差异 |
|----------|--------|-------------|
| PagerDuty / Datadog | 基础设施 告警、应用性能监控 | 聚焦 Agent 运行态（429 / 空检索 / Token），非通用基础设施 |
| Azure SRE Agent / Rootly / Harness | 变更 + 遥测关联、根因 + 置信度 | 不做基础设施 变更归因；做 **Agent 运维根因推理 + 评分可验证** |
| 飞书 / 邮件事件回调 通知器 | 挂了提醒 | **推理链 + 评分 + 飞轮**，不是通知换壳 |
| 通用聊天机器人 助手 | 自然语言问答 | **事件驱动处置手册+ 结构化处置 + 审计** |

更完整的调研矩阵见 [coagent-competitive-analysis.html](./coagent-competitive-analysis.html)。
