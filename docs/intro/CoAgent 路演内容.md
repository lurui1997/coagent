## 1. 一句话定位（One-liner）

**口号：**
> **为 Agent 上生产护航：诊断「错在哪里」，研判「怎么处理」，协同「帮你处理」**

**一句话价值：**
> CoAgent 是面向 AI Agent 进入生产环境后的运行事故场景（运行失败、质量异常、成本失控），提供异常发现、根因诊断、风险决策、受控处置和审计复盘能力的产品。

**一句话差异：**
> 别人把「看见、诊断、处理」拆在不同工具里、还得你自己拼，且大多只停在屏幕上；CoAgent 把三步收进**一个闭环**——从看见问题到帮你动手处理，一体化完成。

---

## 2. 问题（Problem）

**谁在痛：** Agent 上线后，负责判断 / 审批 / 执行处置的人——往往**不是原开发者**（FDE、值班 SRE、客户 IT）。

**四类核心痛点（各配一个具体场景）：**

| 痛点 | 真实场景 | 现在怎么凑合 |
|------|----------|--------------|
| **运行异常难诊断** | 客服 Agent 大促并发翻倍触发 API 429，失败率冲到 **38%** | 跨监控 / Trace / 配置反复排查，还得等原工程师拍板 |
| **质量问题难发现** | RAG Agent 知识库索引延迟，空检索率 **35%**，仍返回幻觉答案 | 基础设施指标全绿，盲目重试根本没用 |
| **成本失控难止损** | 内容 Agent 长文模板让 Token 暴涨 **180%**，超日预算 | 只看得到超支，不知道该限流、降级还是暂停 |
| **处置风险难判断** | 客服 Agent 要在重试 / 降级 / 暂停里选，但不清楚每个动作的影响 | 没有风险依据和权限边界，容易误操作或延误 |

**一句话总结：** 运行异常难诊断、质量问题难发现、成本失控难止损、处置风险难判断——而**最该处理的人恰恰最不懂这个 Agent**。

---

## 3. 现状方案的不足（Why now / Gap）

**为什么是现在（市场拐点）：**
- **62%** 企业已试验 Agent，但近 **2/3 尚未规模化**——生产落地刚起步（McKinsey 2025）。
- 仅 **21%** 具备成熟的 Agent 治理能力（Deloitte 2026，覆盖 24 国 / 3,235 名负责人）。
- Gartner 预测：到 2027 年底 **超 40%** 的 Agentic AI 项目会被取消，主因是成本上升、价值不清、风险失控。

**现有工具的不足（缺口在哪）：**
- 市场不缺单点能力——观测、评测、SRE、安全工具都在快速成熟。
- 但它们**分散在不同产品里**：观测工具以 Trace / Session / Eval 为中心，控制台为开发者设计。
- 企业要自己做数据统一、规则配置、执行适配、跨系统审计，集成成本高。

**一句话缺口：** 不是「没有能力」，而是缺少**低集成成本、Agent 原生、面向事故责任人的一体化处置闭环**。

## 4. 你的方案（Solution）

**主线（建议作为全场记忆点）：** CoAgent 把 Agent 事故处理收敛成三步——

> ### 看见问题 → 诊断问题 → 处理问题

| 三段 | 回答的问题 | 包含能力 |
|------|-----------|----------|
| **① 看见问题** | 发生了什么？ | 接入运行事件与指标（`POST /events`）、幂等去重，把运行失败 / 质量异常 / 成本失控识别为「事故」、多信号汇总 |
| **② 诊断问题** | 为什么、有多大把握？ | 汇总日志 / 指标 / 配置 / 手册 → 根因假设 + 影响判断 + **证据链**；把握度评分（数据·手册·推理）给出「有多大把握」 |
| **③ 处理问题** | 要不要动、怎么安全地动、有没有效？ | 风险分级 → **处置合同**（🟢可执行 / 🟡需确认 / 🔴必须升级）→ 飞书 IM 协同 → 受控处置（重试 / 降级 / 暂停 / 升级 + 人工审批）→ 结果验证 → 审计复盘 → 反馈飞轮 |

**为什么这三段是关键（直接接第 3 块的缺口）：**
> 市场上的工具大多停在 **① 看见**（观测 / Trace），少数能做 **② 诊断**（根因 / 评分），**几乎没有人把 ③ 处理做成「让非原作者也能安全执行」的闭环**。CoAgent 的护城河就在第三段。

**与现有方案的 4 个关键差异（建议每条配一句话上台讲）：**

| 创新点 | 核心机制 | 一句话差异 |
|--------|----------|------------|
| **Agent 原生事故模型** | 把运行失败 / 质量异常 / 成本失控统一为「事故」，关联版本、工具调用、业务影响、责任人 | 别人以 Trace 为中心，我们以**需要处置的事故**为中心 |
| **风险分级的处置合同** | 把证据、反证、动作风险、可逆性、审批要求组合成执行边界：🟢 可执行 / 🟡 需确认 / 🔴 必须升级 | 别人给评分和建议，我们让风险判断**直接约束动作权限** |
| **面向非原作者的 IM 处置** | 把复杂诊断变成飞书里可理解、可确认、可升级的处置卡片 | 别人服务开发者、在控制台调查，我们让**值班人和客户 IT 也能安全决策** |
| **结果验证驱动的学习飞轮** | 记录事故前后指标、人工决策、执行动作、恢复结果，反哺手册和规则 | 别人止于告警 / 修复建议，我们把**「是否恢复」作为闭环终点** |

**最核心的一句话（建议放标题）：**
> 把 Agent 事故判断转化为一份**有证据、有风险边界、有审批权限、有验证标准的「处置合同」**，让非原作者也能安全执行。

## 5. 技术亮点（How it works）

> 以下均为已落地代码（2026-06-28 核对），不是 roadmap。

**整体链路（Orchestrator 编排）：**
`POST /events` 事件接入 → 幂等去重 → 场景路由 → 处置手册引擎跑工具 → 大模型结构化推理 → 把握度评分 → 多模式编排 → 飞书协同 → 审计落库 → SSE 实时推送

**关键技术选型与设计：**

- **真实大模型 + 结构化输出**：用 Pydantic `LLMOutput` 约束模型输出；OpenAI 兼容 API，无 Key / `MOCK_LLM` 时降级模拟；失败重试 1 次，**L0 兜底不伪造 hypothesis**（宁可不答也不编）。
- **把握度评分（核心算法）**：三因子加权——**数据可信度 · 手册匹配度 · 推理一致性** → 总分 → 🟢🟡🔴 分级。这是「有多大把握」的量化来源。
- **处置手册引擎**：JSON 单一配置源，`(type, symptom) → playbook` 路由；模拟工具 `metrics / config / ops_playbook`。
- **模型热切换** `ModelRouter`：按 playbook / 风险分级 / 成本策略选模型（如 429 走 fallback、成本场景强制降级小模型）。
- **运行时纠偏**：基于评分与反馈生成提示词 / 参数修正建议，可经 webhook 回调下发。
- **企业级审计**：全链路 SQLite 持久化、`operator` 操作人标识、审计动作链、`/admin/audit/export` 支持 JSON / CSV 导出。
- **Ultra 智能编排（已接入 UI）**：
  - 知识图谱 `knowledge_graph`：Agent / 事件 / 根因 / 处置实体关系 + **多跳根因查询**。
  - 历史推演 `historical`：相似事故检索 + **What-if 反事实模拟**（「若当时降并发，评分会变多少」）。
  - 多模式编排 `team_orchestrator`：按事件复杂度自动选 Single / Sub-agent / Multi-Agent / Team。
- **工程化护栏**：超时预算（Tool 3s / LLM 15s / 链 30s）、SSE 实时时间线、只读回放（不重复调大模型）、日统计 UPSERT。

**技术栈：** FastAPI + SQLite + SSE + Jinja2，OpenAI 兼容大模型，飞书 IM 集成。

**护城河（建议一句话）：** 不是某个新模型，而是**「事故→处置合同」的决策机制 + 结果验证飞轮**——越多事故流经系统，手册和风险规则越准。

## 6. Demo / 成果

> 黑客松权重最高的一块，准备好**截图或录屏**，按下面顺序演示最有冲击力。

**镇场数据——三场景 E2E 真跑通：**

| 场景 | 事件 | 评分 | 分级 | 处置动作 |
|------|------|:----:|:----:|----------|
| **S1** 客服限流 | run_fail + rate_limit | **87** | 🟢 可执行 | 一键重试 |
| **S2** RAG 空检索 | run_fail + empty_retrieval | **72** | 🟡 需确认 | 禁止重试 |
| **S3** 内容超预算 | cost_report + over_budget | **58** | 🔴 必须升级 | 升级 @oncall |

> 同一套引擎对三种事故给出**差异化、可解释**的处置——这是最能打动评委的一幕。

**可现场演示的能力清单：**
1. Demo 控制条一键触发 S1 / S2 / S3。
2. 管理台实时审计时间线**逐条滚动**（SSE 推送：工具调用 → 大模型推理 → 评分 → 编排 → 协同）。
3. 评分分解面板：三因子拆解 + 推理链 + 差异化动作按钮。
4. 飞书卡片推送 + 一键重试（需 `FEISHU_*` 凭证；否则展示卡片渲染）。
5. 反馈飞轮 👍/👎 → 统计实时更新。
6. 审计复盘：故障详情 + **只读回放**。
7. Ultra：知识图谱可视化、What-if 推演、编排模式展示。
8. 审计导出 JSON / CSV。

**工程完成度（佐证「不是 PPT 产品」）：**

- 测试套件覆盖 scoring / router / idempotency / replay / correction / model_router / ultra / audit_export。

## 7. 市场 / 用户（Who & Market）

**目标用户（核心：事故发生后负责判断 / 审批 / 执行的人）：**

| 用户 | 典型角色 | 核心诉求 |
|------|----------|----------|
| **Agent 交付团队** | FDE、AI 解决方案商、系统集成商 | 用少量人力维护多客户的多个 Agent |
| **企业 AI 平台 / 运维** | 电商、金融、SaaS 的 AI 平台负责人、SRE | 保障线上 Agent 稳定，控成本风险 |
| **客户 IT / 业务负责人** | 接手第三方 Agent 的客户方 | 看懂处置依据，完成审批或升级 |

> 非优先用户：少量、低风险、由原开发者自己维护的 Agent。

**市场规模信号（用需求侧数据替代 TAM 硬编）：**
- 62% 企业已试验 Agent，规模化生产是下一波必经阶段。
- 仅 21% 有成熟治理 → 治理是公认的空白区。
- 40%+ 项目因风险 / 成本失控被取消 → 痛点有真实的「钱的代价」。

⚠️ 若评委追问具体市场规模（$ TAM/SAM），目前文档没有量化数字，建议要么不给数字、只讲趋势，要么补一个可辩护的自下而上估算（如：目标企业数 × 单客年付费）。

## 8. 商业化 / 影响力（Business / Traction）

**当前的「Traction」= 完成度本身：** 一个真跑通三场景、Basic→纠偏→模型路由→审计→Ultra 全链路落地的可演示系统（黑客松语境下，这就是最强的早期验证）。

⚠️ **商业模式是目前最薄弱的一块——选题文档里没有明确定义。** 下面是建议的**假设**，请你确认后再上台（别当既成事实讲）：

- **可能的计费维度**（ToB Agent Ops 常见）：
  - 按**接入的 Agent 数量** / 监控的事故量计费；
  - 按**席位**（值班 / 审批人）订阅；
  - 平台年费 + 增值模块（Ultra 编排、合规审计导出）。
- **落地切入点假设**：先服务 **Agent 交付团队（FDE / 解决方案商）**——他们最缺「少人维护多客户」的杠杆，付费意愿和紧迫度最高。
- **规模化逻辑**：低集成成本 + Agent 原生 → 接入快；反馈飞轮 → 越用越准 → 切换成本高。

> 答辩建议：商业化别讲太满，给一个清晰的「先服务谁、怎么收第一笔钱」即可，比堆砌大市场更可信。
