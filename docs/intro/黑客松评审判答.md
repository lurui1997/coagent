# CoAgent 黑客松评审判答

**项目：** CoAgent — ToB Agent Ops Copilot  
**日期：** 2026-06-28  
**用途：** 路演答辩、评委问答、团队内部对齐  
**对照基线：** [coagent-design-spec.md](../superpowers/specs/coagent-design-spec.md) · pytest `-m "not live_llm"` · Admin 三 Tab + S1/S2/S3 全流程可跑

> 评委最终看的是：**你是否真的做了一件有价值的事，并且能讲明白。**  
> 下文按评分维度逐项回答，并标注「强项 / 风险 / 答辩话术」。

## 1. 场景创新性（权重 30%）

### 这个问题值不值得做？

**值得。** 痛点真实且正在放大：

| 痛点 | 业务后果 | 现有工具缺口 |
|------|----------|--------------|
| Agent 429 / 空检索等**静默失败** | 客户先发现，而非系统 | PagerDuty / APM 管 infra，不管 Agent 运行态 |
| 不知先**重试、改 prompt 还是换模型** | 翻日志、瞎改配置 | 通用 Chatbot 无 Ops 决策边界 |
| Token **单日超预算**无分级响应 | 烧钱、权责不清 | 成本告警缺「敢不敢动」的把握度 |
| 说不清当时建议了什么 | 合规与复盘风险 | 通知类工具无结构化审计链 |

CoAgent 切入的是 **「Agent 运行态运维」真空层**——不是「服务器挂了」，而是「客服 Bot 被限流」「RAG 空检索导致幻觉」「内容 Bot 烧钱」。

### 它是换个壳，还是有真正新视角？

**有新视角，但需要主动讲清楚，否则容易被误判为换壳。**

| 评委可能的第一印象 | 实际情况 | 如何扭转叙事 |
|-------------------|----------|--------------|
| 「又一个监控 + 大模型写建议」 | 核心是 **处置闭环 + 可验证把握度**，不是告警面板 | 开场 30 秒讲「三件套」，不要先讲技术栈 |
| 「跟 Azure SRE Agent / Datadog 一样」 | 那些做 **infra 变更关联**；我们做 **Agent 层根因链**（限流→不可用、空检索→幻觉、超预算→止血） | 强调「管 Agent 失败/质量/成本，不管 K8s 5xx」 |
| 「大模型自报 confidence 罢了」 | 把握度来自 **三因子公式**（数据·手册·推理），可追问、可校准 | 现场打开 Tab3 展示分解，或口述公式 |

**真正的新视角（答辩主线）：**

```
Agent 异常 → 手册约束根因推理 → 把握度评分（非 LLM 自评）→ 分级处置边界 → 审计飞轮
```

同一类 `run_fail` 事件，S1 评分 87 🟢 建议重试，S2 评分 72 🟡 **禁止盲重试**——这不是换壳，是 **「敢不敢动」由可验证规则定义**。

### 自评

| 项 | 评分倾向 | 说明 |
|----|----------|------|
| 问题价值 | ⭐⭐⭐⭐ | Agent 进业务、运维没跟上，趋势明确 |
| 切入新意 | ⭐⭐⭐⭐ | Agent Ops 层 + 把握度评分，区别于 infra SRE |
| 被质疑风险 | ⚠️ 中等 | 若只演示「触发→出结果」，会像固定剧本 Demo |

**答辩金句：**  
「PagerDuty 告诉你 infra 挂了；CoAgent 告诉你 **Agent 为什么不可信、敢不敢重试、谁该升级**。」

## 2. 产品完成度（权重 25%）

### 关键流程是否跑通了？

**是。** 核心链路已在代码中打通，并有测试覆盖：

```
POST /events 或 Admin 触发 S1/S2/S3
  → 场景路由 (type, symptom) → playbook
  → PlaybookEngine 跑工具 → LLM 结构化推理
  → compute_score 三因子评分 → 🟢🟡🔴 分级
  → SSE 实时审计时间线 → SQLite 持久化
  → S1 管理台模拟重试 + channel_sync 占位
```

**已验证能力（2026-06-28）：**

- Admin 三 Tab：事故总览 · 处置工作台 · 审计复盘
- 三场景 E2E：`demo.sh` 一键触发，预期 S1≈87🟢 · S2≈72🟡 · S3≈58🔴
- 幂等去重、超时预算、回放（不调 LLM）、👍/👎 反馈统计
- **51 passed** 单元测试（scoring / router / idempotency / replay / feishu 等）

### 还是只是演了一个固定路径？

**半真半假——需要诚实表述边界。**

| 真实可运行 | 仍是 Demo 边界 |
|-----------|----------------|
| 流水线通用，三场景共用同一 Orchestrator | 事件 payload 来自 `data/scenarios/s*.json` 预置 |
| LLM 输出可变（有 Key 时走真实 API） | Tools 为 mock（metrics / config / ops_playbook） |
| 评分公式确定性计算，非硬编码展示 | Demo 模式对推理因子有校准披露（Tab 可见） |
| 重试 API 返回 200 并写审计 | 不真正回调客户 Agent 运行时 |

**应对「固定路径」质疑的话术：**

1. **对比演示：** 连续触发 S1 与 S2——同类失败，不同根因链、不同评分、不同处置（能重试 vs 禁止重试）。
2. **展示 SSE 时间线：** 推理链、工具调用、评分分解是实时生成的，不是播片。
3. **承认边界：** 「Hackathon 用 mock 工具保证稳定；赛后第一步是 Webhook 接真实 Agent 平台。」

### 自评

| 项 | 评分倾向 | 说明 |
|----|----------|------|
| 核心流程完整 | ⭐⭐⭐⭐ | 接入→推理→评分→展示→审计闭环 |
| 生产就绪 | ⭐⭐ | 缺真实 Agent 接入、Tools 未接真实系统 |
| Demo 稳定性 | ⭐⭐⭐⭐⭐ | MOCK_LLM + 超时降级，界面不空白 |

## 3. 技术深度（权重 20%）

### 是不是只停留在简单调用 API？

**不是，但 LLM 只是流水线一环。** 工程深度体现在：

| 模块 | 工程逻辑 | 代码位置 |
|------|----------|----------|
| **Orchestrator** | 幂等、async 超时（Tool 3s / LLM 15s / Pipeline 30s）、异常降级、审计写入 | `app/orchestrator.py` |
| **PlaybookEngine** | JSON 单一配置源，改场景不改代码；system_prompt + tool_mocks + consistency_rules | `app/playbooks/engine.py` + `data/ops_playbooks.json` |
| **把握度评分** | 三因子加权 + consistency_clamp + 分级阈值，**非 LLM 自报** | `app/scoring/scorer.py` |
| **LLM 层** | Pydantic LLMOutput 结构化输出、失败重试 1 次、L0 不伪造 hypothesis | `app/models/llm_output.py` · `app/llm/client.py` |
| **实时审计** | SSE 推送 + SQLite 持久化 + 只读回放 | `app/sse.py` · `app/db.py` |
| **协同** | 管理台处置为主;飞书卡片 **规划中** | `app/channels/feishu_im.py`（骨架） |

### 是否存在清晰的 Agent 设计或工程逻辑？

**有，但是「Playbook 驱动的 Ops Copilot」，不是 Multi-Agent  swarm。**

**Agent 设计定位：**

- **角色：** 事件驱动的 Agent 运维副驾驶，不是开放式对话助手
- **决策链：** 手册约束 → 工具取证（mock）→ LLM 根因链 → **规则评分器** → 分级动作
- **编排扩展（Ultra MVP）：** 按场景选择 single / sub / multi / team 模式，输出结构化 team_plan（审计可见，子任务为模拟完成）

**与「简单调 API」的本质区别：**

```
❌ 简单调 API：alert → prompt → 打印建议
✅ CoAgent：event → route → playbook.tools → LLM(JSON) → score(公式) → grade → action + audit
```

**评委可能追问 → 标准答法：**

| 追问 | 答 |
|------|-----|
| 为什么不用 LLM 直接输出 confidence？ | 运维零幻觉容忍；三因子可分解、可测试、可校准（`calibrate_scores.sh`） |
| 评分怎么验证？ | 单元测试 + 三场景预期分；Demo 模式披露校准因子 |
| Multi-Agent 在哪？ | 主链路是 Single Copilot；Ultra 层按复杂度生成 team_plan，Hackathon 重点在处置闭环 |

### 自评

| 项 | 评分倾向 | 说明 |
|----|----------|------|
| 架构合理性 | ⭐⭐⭐⭐ | 分层清晰，配置驱动，有降级策略 |
| Agent 范式深度 | ⭐⭐⭐ | Playbook + 评分器是亮点；Multi-Agent 为 MVP |
| 可测试性 | ⭐⭐⭐⭐ | 51 tests，评分/router 可单测 |

## 4. 商业潜力（权重 15%）

### 谁会用这个产品？

| 用户 | 典型 Agent | 使用场景 |
|------|-----------|----------|
| **中小企业 / OPC** | 客服 Bot、RAG 问答、内容生成 | 无专职 SRE，Agent 挂了只能人工翻日志 |
| **ToB AI 团队 / 交付** | 客户侧部署的 Agent | 需要可审计的处置建议，对客户负责 |
| **运维 / 平台负责人** | 多 Agent 组合 | 要统一看「失败 / 质量 / 成本」三类异常 |

### 为什么会用？

1. **省时间：** 90 秒内从症状到「该不该重试、步骤是什么、把握多大」  
2. **降风险：** 🟡🔴 强制人工确认 / 升级，避免 AI 建议盲执行  
3. **可复盘：** 全链路审计 + 👍/👎 飞轮，满足「当时谁批准的」  
4. **可私有化：** 本地 FastAPI + SQLite，日志不出域（金融/政企友好）

### 是否有初步落地可能？

**有叙事、有 MVP，落地需接真实数据源。**

| 阶段 | 内容 | 状态 |
|------|------|------|
| **Hackathon** | Admin 演示 + mock 工具 + 可选飞书 | ✅ 当前 |
| **赛后 4 周** | 替换 `POST /events` 为客户 Agent 运行时 Webhook | 📅 Spec §16 |
| **3 个月** | 运维手册 RAG、S2/S3 飞书、ModelRouter 成本策略 | 📅 路线图 |
| **商业化** | 按 Agent 数量 / 事件量 SaaS 或私有化部署 | 待定价文档 |

**ROI 话术（可量化）：**

- 一次 Agent 静默失败若造成客诉/漏单，人工成本 >> CoAgent 订阅  
- S3 超预算若早 1 小时止血，Token 费用可省可观比例  
- 审计链降低合规与复盘成本

### 自评

| 项 | 评分倾向 | 说明 |
|----|----------|------|
| 目标用户清晰 | ⭐⭐⭐⭐ | OPC + Agent 进业务的团队 |
| 付费意愿逻辑 | ⭐⭐⭐ | 痛点清楚，需 1–2 个试点客户验证 |
| 落地路径 | ⭐⭐⭐ | Webhook 集成是明确下一步 |

## 5. Demo 表现（权重 10%）

### 有没有在短时间内讲清楚？

**可以。** 推荐 **90 秒结构**：

| 时间 | 内容 | 动作 |
|------|------|------|
| 0–15s | 痛点：Agent 进业务，运维没跟上 | 口头，配合官网/Tab1 |
| 15–30s | 方案：根因推理 + 把握度评分 + 分级处置 | 讲「三件套」 |
| 30–60s | **Live Demo：** 连点 S1 → S2 | Tab2 触发，Tab3 看 🟢 vs 🟡 |
| 60–75s | S3 超预算 🔴 + Tab4 飞轮 | 强调「必须升级」 |
| 75–90s | 差异化 + 落地 | vs Datadog；本地部署；Webhook 下一步 |

**三场景对比表（必背）：**

| 场景 | 问题 | 评分 | 处置 |
|------|------|------|------|
| S1 限流 | 429 → 客服不可用 | 🟢 ~87 | **可重试** |
| S2 空检索 | 索引 lag → 幻觉 | 🟡 ~72 | **禁止盲重试**，改知识库 |
| S3 超预算 | Token 超标 | 🔴 ~58 | **必须升级** @ |

### 问答是否体现了对项目的真实理解？

**准备以下高频题，答出细节即过关：**

| 评委问题 | 要点回答 |
|----------|----------|
| 跟 Datadog / PagerDuty 有什么区别？ | 他们管 infra；我们管 **Agent 失败/质量/成本**，且有 **可验证评分** |
| 把握度怎么算的？ | `100×(0.35×数据+0.35×手册+0.30×推理)`，Tab3 可展开 |
| 为什么 S2 不能重试？ | 根因是空检索→幻觉，重试不解决索引 lag，会重复错误答复 |
| LLM 胡编怎么办？ | 结构化输出 + 评分 consistency_rules clamp；低分 🔴 不自动执行 |
| 是固定剧本吗？ | 事件预置保证 Demo 稳定；流水线通用；LLM 推理链可变；Tools 赛后接真 |
| 飞书干什么？ | S1 🟢 推卡片 + 一键重试，运维不用切控制台 |
| 怎么越用越准？ | 👍/👎 反馈 → 手册/prompt 迭代（飞轮 Tab4） |
| 技术栈？ | Python FastAPI · SQLite · HTMX SSE · OpenAI 兼容 LLM |

## 7. 评委最终问题：你是否真的做了一件有价值的事，并且能讲明白？

### 有价值吗？

**是。** CoAgent 解决的是 Agent 规模化落地后必然出现的 **「运行态不可信、不敢动、无法复盘」** 问题。把握度评分把「AI 建议」变成「可验证的 Ops 决策」，这是 infra 监控和通用 Chatbot 都没闭环的一层。

### 能讲明白吗？

**能，前提是 Demo 顺序要对：**

1. 先讲 **问题**（Agent 静默失败 / 成本失控）  
2. 再讲 **差异**（三因子评分，非 LLM 自吹）  
3. 用 **S1 vs S2** 证明不是固定重试脚本  
4. 用 **S3** 证明分级处置  
5. 最后 **诚实边界** + **赛后 Webhook 落地**

### 团队内部共识（答辩前 5 分钟默念）

> 我们不是又做了一个告警工具。  
> 我们在 Agent 出错或烧钱时，用 **手册约束的根因推理 + 可验证把握度 + 分级处置 + 全链路审计**，回答运维最关心的问题：**该不该动、怎么动、有多大把握。**
